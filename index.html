1.1
Define Web Development and explain its types (frontend, backend, full-stack). Draw and label a Full Stack Architecture.
Definition / Intro
Web Development is the art of building and maintaining websites. It's like building a house: you need someone to design the rooms (Frontend), someone to handle the plumbing and electricity (Backend), and someone who can do both (Full-Stack). The 3 Layers of Web Development

1. FRONTEND (The UI)
↓

Explanation

Web development is a team effort between the UI and the Server. Whether you focus on design (Frontend) or logic (Backend), a Full-Stack understanding is the ultimate goal for any modern developer.

Definition / Intro
An HTML Table is a structured grid used to display data in rows and columns. It follows a strict hierarchy (Table > Row > Data). Alongside tables, understanding Display Properties is critical for layout—they determine if an element takes up a whole line or sits next to its neighbors.
Explanation (The HTML Table Anatomy)
📊 Semantic Table Structure
  • <caption>: The "Title" of the table (SEO friendly).
  • <thead>: Groups the header content (labels). Contains <th>.
  • <tbody>: Groups the main body content (the actual data).
  • <tfoot>: Groups footer content (e.g., "Total" or "Summary").
  • Merged Cells:
    • colspan="2": Merges 2 columns horizontally (stretches wide).
    • rowspan="2": Merges 2 rows vertically (stretches tall).
Explanation (Display Properties Deep-Dive)
📦 Block
  • Starts on New Line.
  • Takes 100% Width.
  • Respects margin/padding.
  • Ex: <div>, <p>, <h1>.
🔡 Inline
  • Sits Next to neighbors.
  • Width = Content size.
  • Cannot set width/height.
  • Ex: <span>, <a>, <b>.
🧩 Inline-Block
  • Sits Next to neighbors.
  • Width = Content size.
  • CAN set width/height.
  • Ex: <button>, <img>, <input>.

2.4
Explain indexing and query optimization in MongoDB with examples.
Definition / Intro
Indexing in MongoDB is a data structure (B-tree) that stores a small portion of the collection's data set in an easy-to-traverse form. Without indexes, MongoDB must perform a Collection Scan (reading every document) to find matches. Indexes allow the database to locate data efficiently, vastly improving Query Optimization and read performance.
Diagram

Without vs With Index

Query: Find User ID=99
↓
NO INDEX: Collection ScanWITH INDEX: Index Scan
↓
Scans 100,000 Documents 🐢Scans 1 Document 🚀
Explanation (Types of Indexes)
  • Single Field Index: An index on a single property (e.g., email).
  • Compound Index: An index on multiple fields (e.g., sorting by age, then by name).
  • Unique Index: Ensures no two documents have the same value for the indexed field (automatically created for the _id field).
Code
javascript
const mongoose = require('mongoose');

const ProductSchema = new mongoose.Schema({
    name: String,
    category: String,
    price: Number,
    sku: { type: String, unique: true } // 1. Creates a Unique Index
});

// 2. Create a Single Field Index (1 for ascending, -1 for descending)
ProductSchema.index({ category: 1 }); 

// 3. Create a Compound Index (Optimizes: find({ category: "Electronics" }).sort({ price: -1 }))
ProductSchema.index({ category: 1, price: -1 });

const Product = mongoose.model('Product', ProductSchema);

// MongoDB Query Optimization Helper
// Using .explain("executionStats") in the mongo shell shows if an index was used.
// db.products.find({ category: "Electronics" }).explain("executionStats");
Conclusion
Proper Indexing is the cornerstone of database performance tuning. While indexes dramatically speed up read operations (queries), they consume memory and slightly slow down write operations, so they must be applied strategically to heavily queried fields.

2.9
Explain advanced Express features: route parameters, query parameters, error handling, and handling async operations (Promise.all).
Definition / Intro
Advanced Express.js features allow for dynamic routing and robust APIs. Route Parameters capture data from the URL path, Query Parameters capture data from the URL query string. A centralized Error Handling Middleware catches crashes gracefully. Promise.all is used to execute multiple asynchronous database queries concurrently to save time.
Extra (Comparison)
Type URL Example Express Syntax Use Case
Route Parameter /users/123 req.params.id Fetching a specific resource by ID
Query Parameter /users?sort=age req.query.sort Filtering, sorting, or pagination
Code
javascript
const express = require('express');
const app = express();

// 1. Route Parameters (Dynamic URLs)
// URL: /api/users/456
app.get('/api/users/:id', (req, res) => {
    const userId = req.params.id; // "456"
    res.json({ user: userId });
});

// 2. Query Parameters (Filtering)
// URL: /api/products?category=shoes&limit=10
app.get('/api/products', (req, res) => {
    const category = req.query.category; // "shoes"
    const limit = req.query.limit; // "10"
    res.json({ category, limit });
});

// 3. Handling Async Operations Concurrently (Promise.all)
app.get('/api/dashboard', async (req, res, next) => {
    try {
        // Run both DB queries at the SAME TIME, rather than waiting for one to finish
        const [users, orders] = await Promise.all([
            UserModel.countDocuments(),
            OrderModel.countDocuments()
        ]);
        res.json({ users, orders });
    } catch (error) {
        next(error); // 4. Forward to Error Handler
    }
});

// 4. Centralized Error Handling Middleware (Must have 4 arguments)
app.use((err, req, res, next) => {
    console.error("Crash logged:", err.message);
    res.status(500).json({ error: "Internal Server Error" });
});
Conclusion
Utilizing Route and Query Parameters allows a single API endpoint to serve infinite data variations. Wrapping asynchronous code in try/catch blocks and routing failures to a centralized Error Handler prevents the Node.js server from crashing in production.

4. Why use it?
  • Less Re-renders: Doesn't refresh the whole page on every keystroke.
  • Smaller Code: Replaces 10 lines of useState with 1 line of register.
5. Diagram
User Input→RHF Internal Cache→Validation (Regex)→Submit

javascript
import { useForm } from "react-hook-form";

function App() {
  const { register, handleSubmit, formState: { errors } } = useForm();
  
  return (

}

3.5
Demonstrate API communication in React using Fetch/Axios with error handling.
Definition / Intro
To display dynamic data from a backend server, React must make HTTP requests. This is commonly done using the native browser Fetch API or a third-party library like Axios. These asynchronous calls are typically placed inside the useEffect hook so the data is fetched when the component mounts.
Diagram

React Data Fetching Lifecycle

Component Mounts
↓
useEffect triggers fetch()
↓
Show Loading Spinner...
↓
Await Promise Resolution
↓
Success? → setData(json)Fail? → setError(msg)
↓
React Re-renders UI with Data
Code
javascript
import React, { useState, useEffect } from 'react';

function UserList() {
  // 1. State for Data, Loading, and Errors
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  // 2. useEffect runs once on mount (empty dependency array [])
  useEffect(() => {
    const fetchUsers = async () => {
      try {
        // 3. Make HTTP Request
        const response = await fetch('https://jsonplaceholder.typicode.com/users');
        
        // Fetch requires manual check for 404/500 errors
        if (!response.ok) throw new Error('Failed to fetch data');
        
        const data = await response.json();
        setUsers(data); // 4. Update State
      } catch (err) {
        setError(err.message); // 5. Catch network errors
      } finally {
        setLoading(false); // 6. Turn off spinner
      }
    };

    fetchUsers();
  }, []);

  // 7. Conditional Rendering based on state
  if (loading) return <p>Loading users...</p>;
  if (error) return <p style={{color:'red'}}>Error: {error}</p>;

  return (
    <ul>
      {users.map(user => <li key={user.id}>{user.name}</li>)}
    </ul>
  );
}
Conclusion
Robust API communication in React requires managing three specific states: Data, Loading, and Error. By wrapping fetch() or axios inside a useEffect hook with a try/catch block, the UI remains responsive and resilient to network failures.

  • Client-Side Nav: Since all the code is already in the browser, the transition happens in milliseconds.

4.3
Explain JWT authentication workflow and role-based access control (RBAC) with protected routes.
Definition / Intro
Role-Based Access Control (RBAC) restricts network access based on the roles of individual users within an enterprise (e.g., Admin vs User). When using JWT, the user's role is encrypted inside the token's payload. Express middleware reads this token, decodes the role, and determines if the user is authorized to access the requested route.
Explanation (The Workflow)
  • Login: User sends credentials. Server returns a JWT containing the user's id and role.
  • Middleware: Every request passes through a verifyToken middleware.
  • Authorization: A secondary middleware checks if the user.role matches the required role for that route.
Diagram

JWT and Role Verification

Client Request→Header: Bearer
↓
verifyToken() Middleware
↓
Valid Token?→No → 401 Unauthorized
↓
Yes
↓
checkRole('Admin') Middleware
↓
Is user.role == Admin?→No → 403 Forbidden
↓
Yes → Access Route
Code
javascript
// 1. JWT Authentication Middleware
const verifyToken = (req, res, next) => {
    const token = req.headers.authorization?.split(" ")[1];
    if (!token) return res.status(401).json({ error: "Access Denied" });

    try {
        req.user = jwt.verify(token, process.env.JWT_SECRET);
        // req.user now contains { id: "123", role: "admin" }
        next();
    } catch (err) {
        res.status(401).json({ error: "Invalid Token" });
    }
};

// 2. Role-Based Access Control (RBAC) Middleware Factory
const requireRole = (requiredRole) => {
    return (req, res, next) => {
        if (req.user.role !== requiredRole) {
            return res.status(403).json({ error: "Forbidden: Admins only" });
        }
        next(); // User has the correct role, allow access
    };
};

// 3. Applying to Protected Routes
app.get('/api/profile', verifyToken, ProfileController.get); 
app.delete('/api/users/:id', verifyToken, requireRole('admin'), UserController.delete);
Conclusion
Combining JWT with RBAC Middleware creates a highly secure and granular application. Authentication (verifyToken) confirms WHO the user is, while Authorization (requireRole) confirms WHAT they are allowed to do.

Demonstrate file upload using Multer and explain challenges in handling large media files.
Definition / Intro
HTML forms submit standard text as application/x-www-form-urlencoded. However, files (images, PDFs) must be submitted as multipart/form-data. Express cannot parse this natively. Multer is a Node.js middleware exclusively used for handling multipart form data, specifically for file uploads.

Multer File Upload Flow

React
↓
FormData() Append→POST multipart/form-data
↓
Express Router
↓
Multer Middleware
↓
Saves file to /uploads folder→Attaches data to req.file
↓

const express = require('express');
const multer  = require('multer');
const path = require('path');
const app = express();

// 1. Configure Multer Storage Engine
const storage = multer.diskStorage({
    destination: function (req, file, cb) {
        cb(null, './uploads/') // Save location
    },
    filename: function (req, file, cb) {
        // Create unique filename: timestamp-originalName
        cb(null, Date.now() + '-' + file.originalname)
    }
});

    storage: storage,
    limits: { fileSize: 5 * 1024 * 1024 } 
});

    });
});

  • Storage Costs: Storing media on the application server (e.g., Heroku) is expensive and often wiped on restart. Solution: Upload directly to AWS S3 buckets.

4.7
Differentiate authentication vs authorization and analyze security risks with mitigation.
Definition / Intro
Authentication (AuthN) is verifying who someone is (Identity). Authorization (AuthZ) is verifying what they are allowed to do (Permissions). A complete security system requires both. For example, logging in is Authentication; being denied access to an Admin dashboard is Authorization.
Extra (Comparison)
Comparison Authentication Authorization
Definition Verifies Identity Verifies Permissions/Roles
Analogy Checking ID at a concert door Checking if ticket is VIP or General
Mechanisms Passwords, Biometrics, 2FA RBAC, ABAC, JWT Payloads
HTTP Status 401 Unauthorized (Unauthenticated) 403 Forbidden (Lacks privileges)
Explanation (Security Risks & Mitigations)
  • Risk 1: SQL/NoSQL Injection — Attackers manipulate DB queries via inputs.
    Mitigation: Use Mongoose (ODM) which sanitizes inputs, and avoid raw string queries.
  • Risk 2: XSS (Cross-Site Scripting) — Attackers inject malicious JavaScript into the UI.
    Mitigation: React automatically escapes variables in JSX. Never use dangerouslySetInnerHTML.
  • Risk 3: Password Breaches — Storing plain-text passwords.
    Mitigation: ALWAYS hash passwords using bcrypt with a salt round before saving to the DB.
  • Risk 4: Token Theft — Attackers steal JWTs from LocalStorage.
    Mitigation: Store JWTs in HttpOnly cookies which cannot be accessed by client-side JS.

↓
AuthZ: Access Key→Enter Room

javascript

  const salt = await bcrypt.genSalt(10);
  return await bcrypt.hash(pwd, salt);
};
Conclusion
A secure application must establish identity (Authentication) before enforcing rules (Authorization). Recognizing common attack vectors like XSS and applying mitigations like bcrypt hashing is essential for protecting user data.
4.8
Explain asynchronous data flow in applications (likes/comments scenario) and performance benefits.
Definition / Intro
When you click "Like" on Instagram, the heart turns red instantly. It doesn't freeze the app while it talks to the server. This is called an Optimistic UI Update. The frontend "optimistically" assumes the server will say yes, so it updates the screen immediately. Meanwhile, it sends the actual request to the backend in the background (asynchronously). This makes the app feel incredibly fast.
Diagram

Optimistic Update (Like Button)

User clicks 'Like'
↓
1. Update UI Instantly (count + 1)
↓
2. Send background Axios Request
↓
Wait for Backend...
↓
Success? Do nothingFail? Revert UI (count - 1)
Code
javascript
import React, { useState } from 'react';
import axios from 'axios';

function LikeButton({ postId, initialLikes }) {
    const [likes, setLikes] = useState(initialLikes);
    const [liked, setLiked] = useState(false);

    const handleLike = async () => {
        // 1. OPTIMISTIC UPDATE: Update UI immediately
        setLikes(likes + 1);
        setLiked(true);

        try {
            // 2. Asynchronous Background Request
            await axios.post(`/api/posts/${postId}/like`);
            // Success: Do nothing, UI is already correct
        } catch (error) {
            // 3. REVERT on Failure (Rollback)
            setLikes(likes); // Revert to old count
            setLiked(false);
            alert("Failed to like post. Please check your connection.");
        }
    };

    return (
        <button onClick={handleLike} disabled={liked}>
            👍 {likes} Likes
        </button>
    );
}
Conclusion
Asynchronous data flow utilizing Optimistic UI updates masks network latency completely. By updating the state immediately and executing HTTP requests in the background, applications feel incredibly fast and responsive, rolling back only if an error occurs.
